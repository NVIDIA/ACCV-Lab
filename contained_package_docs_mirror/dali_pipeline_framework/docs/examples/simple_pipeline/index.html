Simple Pipeline (Introductory Example)

This example demonstrates the basic building blocks of the DALI pipeline framework using synthetic data – no external datasets are required to run it. It is intended as a starting point for understanding the core design before diving into the more complex examples (2D Object Detection Pipeline, StreamPETR Pipeline).

The use of synthetic data keeps the example self-contained while still demonstrating the important concepts. The included code snippets/files are best read in order: data provider, pipeline setup, and then entry point.

Concepts Covered

  • Sample data structure: Defining a hierarchical SampleDataGroup blueprint representing the training data format. In this example the blueprint contains a cameras array with per-camera image and annotation/label fields, as well as sample-level scene_label and scene_label_as_str fields. For details on the blueprint concept and the SampleDataGroup class in general, please also see Data Format: Sample Data Group.

  • Data provider: Subclassing DataProvider to provide the defined data structure blueprint as well as individual training samples matching this structure. In this example, those samples are generated synthetically.

  • Input callable: Wrapping the data provider with ShuffledShardedInputCallable to provide shuffled samples to the pipeline.

  • Processing steps: Chaining built-in processing steps (PhotoMetricDistorter, ImageRange01Normalizer, AnnotationElementConditionEval) to define the processing pipeline.

  • Access modifier wrappers: Applying photometric augmentation independently per camera via DataGroupArrayInPathElementsAppliedStep.

  • Pipeline assembly: Creating a PipelineDefinition, building the DALI pipeline, and iterating batches through DALIStructuredOutputIterator (which returns nested dicts).

  • Output handling: Obtaining the output (which is dumped using TensorDumper in this simple example and would be used for training in a real-world use-case).

Important

You can run the example using the script packages/dali_pipeline_framework/examples/simple_full_pipeline/run_example.py.

No dataset preparation is needed – all data is generated synthetically.

See also

  • Introduction for the overall pipeline framework design and a more detailed discussion on the individual concepts.

  • 2D Object Detection Pipeline and StreamPETR Pipeline for full-scale pipeline examples using NuScenes data.

  • Flexible Step Implementation for a tutorial on implementing custom processing steps.

Data Provider

The data provider is the boundary between use-case-specific data loading and the reusable pipeline framework. It defines the sample contract once as a SampleDataGroup blueprint, then returns samples that follow this contract. When designing a pipeline for a specific use-case, this is the place to decide the input data format and to implement the data loading logic.

Note

The output data format for the overall pipeline is a consequence of the input data format and the processing steps applied to it, so that it does not need to be defined explicitly.

 26# @NOTE
 27# These mappings demonstrate how a `SampleDataGroup` data field can accept human-readable strings while storing
 28# numeric IDs in the DALI pipeline. When a mapped field is assigned a string key, the framework converts it to
 29# the corresponding integer value. This is e.g. useful for class labels (e.g. "vehicle", "pedestrian", "cyclist").
 30CAMERA_LABEL_TO_ID = {
 31    "vehicle": 1,
 32    "pedestrian": 3,
 33    "cyclist": 6,
 34    "construction": 8,
 35}
 36SCENE_LABEL_TO_ID = {
 37    "indoor": 0,
 38    "urban": 1,
 39    "highway": 2,
 40    "tunnel": 3,
 41}
 42CAMERA_ID_TO_LABEL = {label_id: label for label, label_id in CAMERA_LABEL_TO_ID.items()}
 43SCENE_ID_TO_LABEL = {label_id: label for label, label_id in SCENE_LABEL_TO_ID.items()}

 46# @NOTE
 47# A data provider is responsible for two things:
 48#   - describing the sample data structure via a `SampleDataGroup` blueprint
 49#   - returning a filled sample for a requested sample index
 50class ExampleDataProvider(DataProvider):
 51    '''Data provider that generates synthetic multi-camera data for demonstration purposes.
 52
 53    Each sample consists of:
 54      - An array of cameras, each containing a synthetic image and an annotation
 55        with a string label that is mapped to an integer id by
 56        :class:`SampleDataGroup`.
 57      - A global scene label shared across all cameras in the sample, also
 58        provided as a string and converted via a user-defined mapping.
 59
 60    Images are deterministic color gradients so that the output is reproducible
 61    and visually distinguishable across cameras and samples.
 62    '''
 63
 64    def __init__(
 65        self,
 66        num_samples: int = 8,
 67        image_height: int = 64,
 68        image_width: int = 64,
 69        num_cameras: int = 2,
 70        seed: int = 42,
 71    ):
 72        '''
 73
 74        Args:
 75            num_samples: Number of samples available in this provider.
 76            image_height: Height of the generated images in pixels.
 77            image_width: Width of the generated images in pixels.
 78            num_cameras: Number of cameras per sample.
 79            seed: Base seed used for deterministic data generation.
 80        '''
 81
 82        self._num_samples = num_samples
 83        self._image_height = image_height
 84        self._image_width = image_width
 85        self._num_cameras = num_cameras
 86        self._seed = seed
 87        self._camera_label_names = tuple(CAMERA_LABEL_TO_ID)
 88        self._scene_label_names = tuple(SCENE_LABEL_TO_ID)
 89
 90        # @NOTE
 91        # The blueprint contains the field names, nesting, and DALI data types, but no actual sample values.
 92        # It also carries the optional string-to-ID mappings used when values are assigned to mapped fields.
 93        # It is cached because it is reused both for reporting the input format and for creating empty samples
 94        # with the same format in `get_data()`.
 95        self._blueprint = self._build_blueprint(num_cameras)
 96
 97    @override
 98    def get_data(self, sample_index: int) -> SampleDataGroup:
 99        # @NOTE
100        # `DataProvider.get_data()` is called by the input callable to obtain one sample by index. The returned
101        # `SampleDataGroup` must match `sample_data_structure` and contain the actual values for that sample.
102
103        # @NOTE
104        # Start each sample from an empty copy of the blueprint. This keeps the data format fixed while
105        # allowing the actual values to be filled independently for each requested sample.
106        sample = self._blueprint.get_empty_like_self()
107        rng = np.random.RandomState(self._seed + sample_index)
108
109        # @NOTE
110        # The synthetic images are deterministic for a given sample index. This makes the example easy to
111        # inspect while still providing different values across samples.
112        base_color = rng.randint(80, 256, size=3).astype(np.float32)
113
114        # @NOTE
115        # Fill the array field "cameras". Each camera entry is a data group with an image and an annotation
116        # subgroup. Here, the label field is mapped to an integer ID on assignment. String fields can also be
117        # passed through directly, as shown below with `scene_label_as_str`.
118        for cam_idx in range(self._num_cameras):
119            image = self._generate_image(base_color, cam_idx)
120            sample["cameras"][cam_idx]["image"] = image
121            label_name_as_str = self._camera_label_names[rng.randint(len(self._camera_label_names))]
122            sample["cameras"][cam_idx]["annotation"]["label"] = label_name_as_str
123
124        # @NOTE
125        # The same scene label is stored twice to show the difference between a mapped integer field and a raw
126        # string field. `scene_label` is converted via `SCENE_LABEL_TO_ID`, while `scene_label_as_str` is passed
127        # through as string data.
128        scene_label_name_as_str = self._scene_label_names[rng.randint(len(self._scene_label_names))]
129        sample["scene_label"] = scene_label_name_as_str
130        sample["scene_label_as_str"] = scene_label_name_as_str
131
132        return sample
133
134    @override
135    def get_number_of_samples(self) -> int:
136        # @NOTE
137        # `DataProvider.get_number_of_samples()` tells the input callable how many samples are available for
138        # indexing, shuffling, and epoch-length computation.
139        return self._num_samples
140
141    @property
142    @override
143    def sample_data_structure(self) -> SampleDataGroup:
144        # @NOTE
145        # `DataProvider.sample_data_structure` is the provider's format contract. The pipeline asks for this
146        # blueprint before execution to infer the input and output formats. Return an (empty) copy so callers
147        # cannot accidentally fill or mutate the cached blueprint.
148        return self._blueprint.get_empty_like_self()
149
150    def _generate_image(self, base_color: np.ndarray, cam_idx: int) -> np.ndarray:
151        '''Create a synthetic HxWx3 uint8 image.
152
153        Both cameras in a sample share the same ``base_color`` but use
154        different gradient directions (horizontal for camera 0, vertical for
155        camera 1).  This makes the effect of independent per-camera photometric
156        augmentation clearly visible in the output.
157        '''
158        h, w = self._image_height, self._image_width
159
160        # @NOTE
161        # Use different gradient directions for neighboring cameras so that the images are visually distinguishable.
162        if cam_idx % 2 == 0:
163            gradient = np.linspace(0, 1, w, dtype=np.float32)
164            gradient = np.tile(gradient[np.newaxis, :], (h, 1))
165        else:
166            gradient = np.linspace(0, 1, h, dtype=np.float32)
167            gradient = np.tile(gradient[:, np.newaxis], (1, w))
168
169        image = np.stack(
170            [
171                np.clip(gradient * base_color[0], 0, 255),
172                np.clip(gradient * base_color[1], 0, 255),
173                np.clip(gradient * base_color[2], 0, 255),
174            ],
175            axis=-1,
176        ).astype(np.uint8)
177
178        return image
179
180    @staticmethod
181    def _build_blueprint(num_cameras: int) -> SampleDataGroup:
182        '''Construct the hierarchical SampleDataGroup blueprint.
183
184        Structure::
185
186            root
187            +-- cameras (array of ``num_cameras`` data group fields)
188            |   +-- 0
189            |   |   +-- image: UINT8
190            |   |   +-- annotation
191            |   |       +-- label: INT32, with string-to-id mapping
192            |   +-- 1
193            |       +-- ...
194            +-- scene_label: INT32, with string-to-id mapping
195            +-- scene_label_as_str: STRING
196        '''
197        # @NOTE
198        # Build the annotation subgroup first. The `label` field is stored as INT32, but the mapping allows the
199        # provider to assign strings such as "vehicle" or "pedestrian" when filling samples.
200        annotation = SampleDataGroup()
201        annotation.add_data_field(
202            "label",
203            types.DALIDataType.INT32,
204            mapping=CAMERA_LABEL_TO_ID,
205        )
206
207        # @NOTE
208        # A camera is represented as a data group containing the image and the nested annotation group. The
209        # group is later reused as the template for every element of the "cameras" array field.
210        cam = SampleDataGroup()
211        cam.add_data_field("image", types.DALIDataType.UINT8)
212        cam.add_data_group_field("annotation", annotation)
213
214        # @NOTE
215        # The root sample contains the multi-camera array and two sample-level scene labels. `scene_label` is a
216        # mapped INT32 field; `scene_label_as_str` is a STRING field that shows how string data can pass through
217        # the pipeline directly.
218        root = SampleDataGroup()
219        root.add_data_group_field_array("cameras", cam, num_cameras)
220        root.add_data_field(
221            "scene_label",
222            types.DALIDataType.INT32,
223            mapping=SCENE_LABEL_TO_ID,
224        )
225        root.add_data_field(
226            "scene_label_as_str",
227            types.DALIDataType.STRING,
228        )
229
230        return root

Pipeline Setup

The pipeline setup connects the data provider to the pipeline, composes the processing steps for the pipeline, and wraps the resulting DALI pipeline as a structured iterator. In this example, it also demonstrates how an access modifier wrapper is used to apply photometric augmentation independently to each camera while allowing the other processing steps to operate on the full sample.

Note

The use of per-camera photometric augmentation illustrates the distinction between consistent and independent processing: related fields can be grouped and transformed consistently within a selected sub-tree of the input data structure, while different selected sub-trees can receive independent randomization. See Pipeline & Processing Steps and the API documentation of PipelineStepBase and GroupToApplyToSelectedStepBase for a more detailed discussion of this approach.

 27def setup_simple_pipeline(
 28    batch_size: int = 2,
 29    num_samples: int = 8,
 30    num_cameras: int = 2,
 31    image_height: int = 64,
 32    image_width: int = 64,
 33    device_id: int = 0,
 34    seed: int = 42,
 35) -> DALIStructuredOutputIterator:
 36    '''Build and return a ready-to-iterate DALI pipeline for the simple example.
 37
 38    Args:
 39        batch_size: Number of samples per batch.
 40        num_samples: Total number of samples available.
 41        num_cameras: Number of cameras per sample.
 42        image_height: Height of the generated images in pixels.
 43        image_width: Width of the generated images in pixels.
 44        device_id: GPU device id.
 45        seed: Random seed for augmentation reproducibility.
 46
 47    Returns:
 48        An iterator (drop-in replacement for a PyTorch DataLoader) that yields
 49        nested dicts with the processed data.
 50    '''
 51
 52    # ===== Input data handling =====
 53
 54    # @NOTE
 55    # Goal: Create an input callable which provides one sample at a time to DALI's external source. The
 56    # provider defines the sample structure and fills that structure with data on request. This example uses a
 57    # provider that generates synthetic samples so it can run without dataset preparation; in a real use case,
 58    # the same setup pattern can be used with a provider that loads the actual training data.
 59    data_provider = ExampleDataProvider(
 60        num_samples=num_samples,
 61        image_height=image_height,
 62        image_width=image_width,
 63        num_cameras=num_cameras,
 64        seed=seed,
 65    )
 66
 67    # @NOTE
 68    # The general `ShuffledShardedInputCallable` wraps the provider and adds common input behavior: batching,
 69    # shuffling, and optional sharding for multi-GPU setups. This keeps the provider focused on describing and
 70    # returning individual samples.
 71    input_callable = ShuffledShardedInputCallable(
 72        data_provider,
 73        batch_size,
 74        shuffle=True,
 75        seed=seed,
 76    )
 77
 78    # ===== Define processing steps =====
 79
 80    # @NOTE
 81    # The processing steps are defined as separate objects and then composed into a pipeline. This mirrors the
 82    # larger examples, but uses only a small set of steps to avoid bloating the example with unnecessary complexity.
 83
 84    # @NOTE
 85    # Step 1 -- Photometric augmentation for images. `PhotoMetricDistorter` searches for image fields with the
 86    # configured name and applies random brightness, contrast, saturation, and hue changes.
 87    #
 88    # When used directly on the full sample, the same random decisions would be shared across matching fields (i.e. the images of
 89    # all cameras in the sample). In the next step, we wrap it with an access modifier wrapper to apply it independently per
 90    # camera.
 91    photometric_step = PhotoMetricDistorter(
 92        image_name="image",
 93        min_max_brightness=[-32.0, 32.0],
 94        min_max_contrast=[0.5, 1.5],
 95        min_max_saturation=[0.5, 1.5],
 96        min_max_hue=[-18.0, 18.0],
 97        prob_brightness_aug=1.0,
 98        prob_contrast_aug=1.0,
 99        prob_saturation_aug=1.0,
100        prob_hue_aug=1.0,
101        prob_swap_channels=0.0,
102        enforce_process_on_gpu=True,
103    )
104
105    # @NOTE
106    # The access modifier wrapper applies the wrapped step separately to each element of the "cameras" array.
107    # Each camera therefore receives its own random augmentation.
108    #
109    # More generally, access modifiers are useful whenever related fields form a sub-tree that should be
110    # processed consistently, while other such sub-trees should be randomized independently. For example, a
111    # spatial image transform may need to update an image, its segmentation mask, projected points or bounding
112    # boxes, and the corresponding projection matrix with the same sampled transformation. Grouping all fields
113    # for one camera into one selected sub-tree keeps these correspondences consistent, while still allowing a
114    # different random transformation for each camera. Please see the API documentation for the available access modifier
115    # wrappers.
116    independent_photometric_per_camera = DataGroupArrayInPathElementsAppliedStep(
117        photometric_step,
118        path_to_array_to_apply_to="cameras",
119    )
120
121    # @NOTE
122    # Step 2 -- Normalize all images (i.e. all fields named "image" in the data structure) to the [0, 1] float range.
123    image_normalizer = ImageRange01Normalizer("image")
124
125    # @NOTE
126    # Step 3 -- Evaluate a condition on the per-camera annotations. For each "annotation" data group, the
127    # expression creates a new "is_active" field alongside "label". Keeping `label` in the output makes it easy
128    # to compare the original mapped label with the derived flag when inspecting the result.
129    condition_eval = AnnotationElementConditionEval(
130        annotation_field_name="annotation",
131        condition="is_active = label >= 3 and label < 8",
132        remove_data_fields_used_in_condition=False,
133    )
134
135    # @NOTE
136    # Store the processing steps in execution order. This list defines the steps which are performed in the
137    # pipeline and their order.
138    processing_steps = [
139        independent_photometric_per_camera,
140        image_normalizer,
141        condition_eval,
142    ]
143
144    # ===== Pipeline definition =====
145
146    # @NOTE
147    # Define the pipeline from the input callable and the processing steps.
148    #
149    # `check_data_format` controls type/format checks inside the DALI pipeline. It is useful to enable during
150    # development, when adding custom steps, or when changing the input data structure, as it catches mistakes
151    # such as assigning a field with the wrong DALI type. After the pipeline and data format have been
152    # validated, it is usually disabled in production to avoid the additional runtime overhead. We keep it
153    # enabled here because this small tutorial benefits from clearer errors.
154    pipeline_def = PipelineDefinition(
155        input_callable,
156        preprocess_functors=processing_steps,
157        check_data_format=True,  # Enable during development/debugging, disable in production.
158        print_sample_data_group_format=True,  # This is useful to inspect the data format after each step (for debugging).
159    )
160
161    # @NOTE
162    # Infer the output data format blueprint. The resulting `SampleDataGroup` blueprint has the same structure and field types as
163    # the pipeline output and is (further down in the code) used to reconstruct nested dictionaries from DALI's flat output
164    # sequence.
165    output_data_structure = pipeline_def.check_and_get_output_data_structure()
166
167    # @NOTE
168    # Create the underlying DALI `Pipeline` object.
169    pipe = pipeline_def.get_dali_pipeline(
170        enable_conditionals=True,  # <- this needs to be enabled for the pipelines build with this framework.
171        batch_size=batch_size,
172        num_threads=2,
173        device_id=device_id,
174        seed=seed,
175        py_num_workers=2,
176        py_start_method="spawn",
177    )
178
179    # @NOTE
180    # Build the DALI pipeline.
181    pipe.build()
182
183    # ===== Wrap as iterator =====
184
185    # @NOTE
186    # Wrap the DALI pipeline so consumers receive nested dictionaries matching the `SampleDataGroup` structure
187    # instead of the flat tuple returned by the raw DALI pipeline. The wrapper can be used as a drop-in
188    # replacement for a PyTorch DataLoader in training code.
189    num_batches = input_callable.length
190    result_iterator = DALIStructuredOutputIterator.CreateAsDataLoaderObject(
191        num_batches, pipe, output_data_structure
192    )
193
194    return result_iterator

 43def _print_batch(batch: dict, batch_idx: int):
 44    '''Print a human-readable summary of one batch (nested dict of tensors).'''
 45    print(f"\n===== Batch {batch_idx} =====")
 46
 47    # @NOTE
 48    # The `DALIStructuredOutputIterator` reconstructs the nested sample structure from the flat DALI output.
 49    # The batch can therefore be accessed like the original `SampleDataGroup` hierarchy: cameras -> annotation
 50    # -> label/is_active, plus sample-level scene label fields.
 51    cameras = batch["cameras"]
 52    num_cameras = len(cameras)
 53    for cam_idx in range(num_cameras):
 54        cam = cameras[cam_idx]
 55        img = cam["image"]
 56        annotation = cam["annotation"]
 57        label = annotation["label"]
 58        is_active = annotation["is_active"]
 59
 60        print(
 61            f"  Camera {cam_idx}:"
 62            f"  image shape={tuple(img.shape)}  dtype={img.dtype}"
 63            f"  |  label={label.flatten().tolist()}"
 64            f"  is_active={is_active.flatten().tolist()}"
 65        )
 66
 67    scene_label = batch["scene_label"]
 68    scene_label_as_str = batch["scene_label_as_str"]
 69    print("  Scene label:" f"  mapped={scene_label.flatten().tolist()}" f"  original={scene_label_as_str}")

 78    # `setup_simple_pipeline()` performs all framework-specific setup: provider creation, input callable
 79    # wrapping, processing-step composition, DALI pipeline build, and output iterator wrapping.
 80    print("Setting up the simple pipeline...")

 82    data_iter = iter(data_loader)

 85    # @NOTE
 86    # Iterate like a regular PyTorch DataLoader. The example resets the iterator on `StopIteration` so it can
 87    # run for a fixed number of demonstration iterations even if the small synthetic dataset is exhausted.
 88    for i in range(num_iterations):
 89        try:
 90            batch = next(data_iter)
 91        except StopIteration:
 92            data_iter.reset()
 93            batch = next(data_iter)
 94
 95        _print_batch(batch, i)
 96
 97        # ===== Dump using TensorDumper (for illustration purposes) =====
 98
 99        # @NOTE
100        # Dump the structured batch. Images are overridden to be written as RGB image files, while the scalar
101        # labels and nested metadata are dumped as JSON-compatible values.
102        dumper.add_tensor_data(
103            "output",
104            batch,
105            TensorDumper.Type.JSON,
106            dump_type_override={"image": TensorDumper.Type.IMAGE_RGB},
107        )
108        dumper.dump()
109
110        print(f"\nDumped to: {dump_dir}")

Expected Output

Running the example prints a short summary for each processed batch, including per-camera image shapes, mapped annotation labels, derived is_active flags, and the sample-level scene labels. Image fields are dumped as RGB image files, while the remaining JSON-compatible values and metadata are stored in the dump for inspection.
